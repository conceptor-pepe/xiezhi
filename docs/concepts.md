# Concepts

## Spec

The long-term truth of the system: behavior, contracts, invariants, and data semantics.

## Change

A single change unit describing why the system is changing, what is changing, how it will be implemented, and how it will be verified.

## Delta

The explicit difference between the current spec and the new intended behavior.

## Gate

A required checkpoint that must pass before work can move forward.

## Verify

The validation step that checks code, docs, contracts, testing, and audit artifacts.

## Archive

The step where a completed change is folded back into long-term specs and recorded as completed work.

## Format Source

The canonical format for each artifact should come from the repository's real workflow rules and established design documents, not from generic template wording.
